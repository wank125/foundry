## Data Lineage - 数据血缘追踪

### 核心概念

Data Lineage 是一个交互式工具，提供数据在 Foundry 平台中流动的整体视图。通过数据血缘，您可以：

- 轻松查找和发现数据集
- 通过强大的界面探索管道
- 与团队成员协作

![Data Lineage 流动动画](https://www.palantir.com/docs/resources/foundry/data-lineage/data-lineage-flow.gif)

### 主要功能

| 功能 | 描述 |
|------|------|
| **搜索发现** | 使用 Project、表和列名搜索数据集 |
| **管道探索** | 展开或隐藏数据集的祖先和后代 |
| **可视化** | 通过着色可视化管道（如着色过期数据集） |
| **详细信息** | 查看架构、最后构建时间、生成代码 |
| **快照共享** | 创建管道快照与其他用户共享 |

### 访问 Data Lineage

1. 从工作区导航栏选择 "Data Lineage"
2. 或使用快捷键 `CMD + J` / `CTRL + J` 搜索

2. 选择导出格式（PNG/SVG）
3. 保存到本地

## Access Control - 访问控制

### 核心概念

Palantir 安全模型支持：

| 特性 | 描述 |
|------|------|
| **严格执行** | 确保用户只能访问被授权的数据 |
| **精细控制** | 支持灵活的访问控制粒度 |
| **透明性** | 用户可查看谁有权访问什么资源 |
| **可用性** | 用户可自信地管理和理解访问控制 |

### 安全模型架构

### 访问控制类型

#### 1. 强制访问控制 (Mandatory Controls)

| 控制类型 | 说明 | 示例 |
|---------|------|------|
| **Organizations** | 组织之间的严格隔离 | 财务部无法访问 HR 数据 |
| **Markings** | 敏感数据标记 | PII、PHI、机密数据 |

#### 2. 自由裁量访问控制 (Discretionary Controls)

| 权限级别 | 操作 |
|---------|------|
| **View** | 查看资源 |
| **Edit** | 编辑资源 |
| **Manage** | 管理权限和设置 |
| **Own** | 完全控制，包括删除 |

### 数据标记 (Markings)

#### 常见标记类型

| 标记 | 描述 | 示例数据 |
|------|------|---------|
| **PII** | 个人身份信息 | 姓名、地址、身份证号 |
| **PHI** | 受保护健康信息 | 医疗记录、诊断信息 |
| **Confidential** | 机密商业信息 | 财务报表、战略计划 |
| **Restricted** | 限制访问数据 | 审计结果、安全日志 |

#### 配置标记

1. 在数据集或项目设置中选择 "Markings"
2. 添加适当的标记
3. 配置标记访问权限
4. 保存更改

## Health Checks - 健康检查

### 核心概念

健康检查用于验证管道中数据的质量。一旦数据同步到 Foundry、在管道中转换并按调度定期运行，健康检查可以确保流经管道的数据保持可靠和预期结构。

### 健康检查类型

| 检查类型 | 验证内容 | 示例 |
|---------|---------|------|
| **Job-level** | 作业成功完成 | 输出数据集的作业是否成功 |
| **Build-level** | 构建成功且在预期时间内完成 | 构建是否在 30 分钟内完成 |
| **Freshness** | 数据保持最新 | 数据是否在过去 1 小时内更新 |

### 配置健康检查

#### 在 Pipeline Builder 中配置

1. 打开 Pipeline Builder
2. 选择输出数据集节点
3. 在右侧面板选择 "Health checks"
4. 添加检查规则

#### 常用检查规则

| 规则 | 参数 | 示例 |
|------|------|------|
| **Row count check** | 最小/最大行数 | 行数 > 1000 |
| **Column exists check** | 列名 | 必须包含 "customer_id" |
| **Schema check** | 列类型 | amount 必须是 double |
| **Freshness check** | 最大延迟 | 数据延迟 < 1 小时 |
| **Null check** | 空值比例 | null 值 < 5% |

### 推荐的健康检查

#### 数据集检查

| 检查 | 优先级 | 配置建议 |
|------|--------|---------|
| **行数范围** | 高 | 设置合理的最小/最大行数 |
| **新鲜度** | 高 | 根据业务需求设置最大延迟 |
| **架构验证** | 高 | 验证关键列存在且类型正确 |
| **空值检查** | 中 | 检查关键字段的空值比例 |
| **唯一性检查** | 中 | 验证主键唯一性 |

#### 管道检查

| 检查 | 优先级 | 配置建议 |
|------|--------|---------|
| **构建成功率** | 高 | 监控作业失败率 |
| **构建时长** | 中 | 检测异常长的构建时间 |
| **资源使用** | 中 | 监控 CPU/内存使用 |

## Data Health - 数据健康监控

### 核心概念

Data Health 应用提供监控视图和仪表板，用于大规模监控管道健康状况。

### Monitoring Views（监控视图）

#### 创建监控视图

1. 打开 Data Health 应用
2. 点击右上角 "New monitoring view"
3. 配置监控范围：
   - 选择 Projects
   - 选择数据集或规则
4. 保存监控视图

![Monitoring Views](https://www.palantir.com/docs/resources/foundry/monitoring-views/overview)

#### 监控视图功能

| 功能 | 描述 |
|------|------|
| **分组监控** | 按项目、管道或业务逻辑分组 |
| **聚合视图** | 查看整体健康状况 |
| **趋势分析** | 监控健康趋势变化 |
| **告警配置** | 设置告警条件和通知 |

### 数据期望 (Data Expectations)

#### 配置数据期望

数据期望定义数据应满足的规则：

```python
# 示例：销售数据期望
expectations = {
    "row_count": "greater_than(1000)",
    "customer_id": "not_null() & unique()",
    "amount": "not_null() & greater_than(0)",
    "order_date": "not_null() & in_past()"
}
```

#### 期望类型

| 期望类型 | 说明 | 示例 |
|---------|------|------|
| **Schema expectations** | 架构期望 | 列存在、类型正确 |
| **Data quality expectations** | 数据质量 | 空值、唯一性、范围 |
| **Business rules** | 业务规则 | 金额 > 0、日期在范围内 |

### 告警和通知

#### 配置告警

1. 在健康检查或监控视图中
2. 选择 "Notifications"
3. 配置告警条件：
   - 检查失败
   - 检查超时
   - 数据异常
4. 选择通知方式：
   - Email
   - Slack
   - PagerDuty

#### 告警最佳实践

| 实践 | 说明 |
|------|------|
| **分级告警** | 严重/警告/信息，不同级别不同处理 |
| **告警聚合** | 相似问题聚合，减少告警疲劳 |
| **告警抑制** | 维护窗口内抑制非关键告警 |
| **自动恢复** | 配置自动重试和恢复机制 |

## Linter - 代码质量检查

### 核心概念

Linter 应用检查 Foundry 状态中的反模式，并建议将资源移至更好状态的方法。可以使用这些建议来降低成本、提高管道稳定性和弹性。

![Linter 应用首页](https://www.palantir.com/docs/resources/foundry/linter/splash.png)

### Linter 术语

| 术语 | 定义 |
|------|------|
| **Resource scope** | 将被 Linter 扫描的资源集合 |
| **Rule scope** | 将针对资源运行的一组规则 |
| **Sweep schedule** | 定义资源范围和规则范围的配置 |
| **Sweep** | 单次扫描调度运行 |
| **Rule** | 针对资源范围评估的预定义逻辑 |
| **Recommendation** | 与特定规则、资源和项目关联的建议 |
| **Fix Proposal** | Linter 创建的实施建议提案 |

### Linter 模式

| 模式 | 目标 | 示例建议 |
|------|------|---------|
| **Pipeline Cost** | 降低计算成本 | 使用增量管道、优化 Spark 配置 |
| **Pipeline Resilience** | 提高稳定性 | 添加健康检查、配置重试逻辑 |
| **Security** | 增强安全性 | 移除不必要的标记、配置访问控制 |
| **Migration** | 迁移到新版本 | 升级到最新 API、弃用旧组件 |

### 扫描调度

#### 配置扫描调度

1. 打开 Linter 应用
2. 创建新的 Sweep schedule
3. 配置：
   - 资源范围（Projects/Spaces）
   - 规则范围（规则集）
   - 扫描频率（每日/每周）
4. 保存调度

#### 扫描结果

扫描完成后，Linter 显示：
- 按影响排序的建议列表
- 每个建议的详细信息
- 预估影响（成本节省、稳定性提升）
- 修复建议

### Linter 规则

#### 常见规则类别

| 类别 | 规则示例 |
|------|---------|
| **性能优化** | 大数据集使用增量管道 |
| **成本优化** | 使用虚拟表而非物化视图 |
| **安全** | 敏感数据应用适当标记 |
| **可维护性** | 代码复杂度检查 |
| **最佳实践** | 使用推荐的 Spark 配置 |

#### PySpark 风格指南

Palantir 提供的 PySpark 风格指南规则：

| 规则 | 建议 |
|------|------|
| **链式表达式** | 链条不超过 5 个语句 |
| **命名约定** | 使用描述性变量名 |
| **注释** | 为复杂逻辑添加注释 |
| **函数长度** | 函数不超过 50 行 |
| **测试覆盖** | 为转换添加单元测试 |

### 推荐和建议

#### 查看推荐

1. 在 Linter 应用中查看扫描结果
2. 筛选推荐：
   - 按影响级别
   - 按类别
   - 按项目
3. 点击推荐查看详细信息

#### 应用修复

对于带修复提案的建议：
1. 审查修复提案
2. 创建分支或提案
3. 合并修复

## 故障排除

### Data Lineage 常见问题

| 问题 | 解决方案 |
|------|---------|
| **图形不更新** | 刷新页面，清除缓存 |
| **无法查看权限** | 检查自己是否有管理权限 |
| **构建历史缺失** | 检查数据集是否有构建记录 |
| **图形加载缓慢** | 缩小图形范围，使用过滤器 |

### Access Control 常见问题

| 问题 | 解决方案 |
|------|---------|
| **无法访问资源** | 检查组织成员资格和权限 |
| **标记未生效** | 验证标记权限配置 |
| **跨组织访问失败** | 确认是否配置了跨组织共享 |
| **审计日志缺失** | 检查审计日志保留策略 |

### Health Checks 常见问题

| 问题 | 解决方案 |
|------|---------|
| **误报失败** | 调整检查阈值，添加数据波动容忍 |
| **检查不执行** | 验证调度配置，检查资源权限 |
| **告警未收到** | 检查通知配置和邮件/Slack 设置 |
| **性能影响** | 优化检查查询，使用采样 |

### Data Health 常见问题

| 问题 | 解决方案 |
|------|---------|
| **监控视图不更新** | 手动触发扫描，检查调度状态 |
| **期望失败** | 验证数据是否符合期望定义 |
| **告警风暴** | 配置告警聚合和抑制规则 |

### Linter 常见问题

| 问题 | 解决方案 |
|------|---------|
| **扫描失败** | 检查资源权限，验证扫描配置 |
| **建议不相关** | 调整规则范围，排除特定项目 |
| **修复提案失败** | 检查分支权限，验证代码库状态 |
